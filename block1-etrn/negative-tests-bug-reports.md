# Негативные проверки и баг-репорты (Блок 1)

## Негативные сценарии

### Н-1. Попытка подписать Т1 после того, как Т2 уже подписан

**Ожидаемое поведение:** Система должна блокировать подписание Т1, если Т2 (водитель) или Т23 (перевозчик) уже подписаны. Сообщение: «Невозможно подписать Т1: документ уже принят перевозчиком».

**Причина:** Нарушение бизнес-процесса — Т1 должен быть подписан до Т2. Если система позволяет подписать Т1 постфактум — это нарушение хронологии документооборота и приведёт к расхождению данных в ГИС ЭПД.

---

### Н-2. Изменение состава груза после подписания Т1

**Ожидаемое поведение:** После подписания Т1 грузоотправителем изменение состава груза (добавление/удаление позиций, изменение веса) должно быть заблокировано. Возможно только:
- Аннулирование текущего документа и создание нового (если Т2 ещё не подписан)
- Оформление дополнительного титула-исправления

**Причина:** Целостность данных. Если груз можно менять после подписания, то перевозчик и грузополучатель получают недостоверную информацию.

---

### Н-3. Подписание Т3 получателем без предварительного Т2/Т23

**Ожидаемое поведение:** Система не должна позволять грузополучателю подписать Т3, если перевозчик не принял груз (Т2/Т23 отсутствуют). Сообщение: «Невозможно подписать Т3: документ не принят перевозчиком».

**Причина:** Логическая ошибка в workflow — груз не может быть доставлен, если он не был принят к перевозке.

---

### Н-4. Отправка в ГИС ЭПД документа с незаполненными обязательными полями

**Ожидаемое поведение:** При попытке отправки документа в ГИС ЭПД система оператора ЭДО должна выполнить XSD-валидацию. Если обязательные поля отсутствуют — отправка блокируется с детальным сообщением: «Ошибка валидации: отсутствует обязательное поле "НаимГруз" в описании груза».

**Причина:** ГИС ЭПД отклонит некорректный документ. Лучше отловить ошибку на стороне оператора ЭДО, чем получить отказ из ГИС ЭПД.

---

### Н-5. Подписание документа КЭП с истекшим сертификатом

**Ожидаемое поведение:** Система проверяет сертификат КЭП перед подписанием. Если сертификат истёк — подписание блокируется с сообщением: «Сертификат электронной подписи недействителен (истёк срок действия)».

---

### Н-6. XSS/инъекция в текстовые поля

**Ожидаемое поведение:** При вводе в поле «НаимГруз» (или любое текстовое поле) значения `<script>alert('xss')</script>` система должна экранировать/санитизировать ввод. При отображении в UI скрипт не выполняется. При формировании XML специальные символы экранированы в соответствии со стандартом XML.

---

### Н-7. Отправка веса груза, превышающего грузоподъёмность ТС

**Ожидаемое поведение:** Если в Т1 указан вес груза 25 000 кг, а в данных ТС указана грузоподъёмность 20 000 кг, система должна показать предупреждение: «Вес груза превышает грузоподъёмность указанного транспортного средства». Пользователь может проигнорировать или исправить.

---

### Н-8. Повторная отправка одного и того же документа в ГИС ЭПД

**Ожидаемое поведение:** Система должна отслеживать состояние отправки. Если документ уже передан в ГИС ЭПД (есть УИД Минтранса), повторная отправка блокируется. Сообщение: «Документ уже зарегистрирован в ГИС ЭПД, УИД: XXX».

---

## Баг-репорт #1

### Заголовок
Система позволяет подписать Т1 после того, как Т2 и Т23 уже подписаны перевозчиком

### Серьёзность
**Critical (P0)** — нарушает целостность бизнес-процесса и хронологию документооборота

### Окружение
- Платформа: 1С:TMS / Веб-интерфейс ЭДО
- Браузер: Google Chrome 125
- Роль: Грузоотправитель (КЭП)

### Шаги воспроизведения
1. Грузоотправитель создаёт черновик Т1, заполняет все поля, но **не подписывает**
2. Перевозчик получает доступ к документу через роуминг ЭДО (баг в разграничении доступа — Т1 не должен быть доступен перевозчику до подписания отправителем)
3. Водитель подписывает Т2 (ПЭП)
4. Перевозчик подписывает Т23 (КЭП)
5. Документ переходит в статус «Принят к перевозке»
6. Грузоотправитель возвращается к черновику и подписывает Т1 (КЭП)

### Фактический результат
- Т1 подписан постфактум
- Временная метка подписания Т1 позже, чем Т2 и Т23
- Документ имеет некорректную хронологию
- При выгрузке в ГИС ЭПД временные метки противоречат друг другу

### Ожидаемый результат
- Система должна блокировать подписание Т1, если Т2 или Т23 уже подписаны
- Сообщение: «Невозможно подписать Т1: документ уже принят перевозчиком»
- Для изменения Т1 после приёмки необходимо аннулировать документ и создать новый

### Приоритет
**High** — требуется немедленное исправление перед запуском в эксплуатацию

### Дополнительно
См. также баг-репорт #2 (связанная проблема разграничения доступа)

## Баг-репорт #2

### Заголовок
XSD-валидация пропускает документ с отсутствующим полем «ИтогМас» (итого масса прописью)

### Серьёзность
**High (P1)** — некорректный документ будет отклонён ГИС ЭПД, что приведёт к задержке оформления перевозки

### Окружение
- API оператора ЭДО: POST /api/v1/etrn/validate
- Версия формата: 5.01

### Шаги воспроизведения
1. Сформировать XML-файл ЭТрН, в котором отсутствует атрибут `ИтогМас` элемента `СвГруз`
2. Отправить XML через API валидации
3. Проверить ответ

### Фактический результат
- API возвращает статус `200 OK`, `"valid": true`
- Документ считается прошедшим валидацию

### Ожидаемый результат
- API возвращает `422 Unprocessable Entity`
- Тело ответа содержит ошибку: `{"field": "СвГруз.ИтогМас", "error": "Обязательное поле отсутствует"}`
- Согласно XSD: `@ИтогМас use="required"`, без этого атрибута документ не соответствует схеме

### Приоритет
**High** — ошибка валидации приведёт к каскадным отказам при передаче в ГИС ЭПД

### Дополнительно
- Аналогичная проблема может затрагивать другие обязательные атрибуты: `ИтогКолМест`, `СпсОпредМас`
- Рекомендуется добавить интеграционный тест на валидацию всех обязательных атрибутов по XSD

## Баг-репорт #3

### Заголовок
Водитель, не указанный в Т1, может подписать Т2 (приём груза)

### Серьёзность
**High (P1)** — позволяет неуполномоченному лицу принять груз, что создаёт юридические риски

### Окружение
- Мобильное приложение водителя
- Роль: Водитель

### Шаги воспроизведения
1. Грузоотправитель создаёт Т1, указывает водителя: Иванов И.И., паспорт XXXX, номер ТС А001АА
2. Водитель Петров П.П. авторизуется в мобильном приложении
3. Петров открывает документ и подписывает Т2 (ПЭП)

### Фактический результат
- Т2 подписан водителем Петровым
- Имя водителя в Т2 (из данных ПЭП) не совпадает с именем в Т1
- Система не выявила несоответствие

### Ожидаемый результат
- Система сверяет данные водителя из КЭП/ПЭП с данными, указанными в Т1
- При несовпадении — блокирует подписание с сообщением: «Водитель не соответствует указанному в Т1»
- Доступ к документу предоставляется только тому водителю, чьи данные указаны в Т1

### Приоритет
**High** — требуется исправление до передачи в эксплуатацию

### Дополнительно
Данная проверка критична для юридической значимости документа и для ответственности за сохранность груза.
