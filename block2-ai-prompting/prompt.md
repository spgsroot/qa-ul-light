# Промпт для LLM: Чек-лист тестирования формы создания заявки на перевозку

## Системный промпт (system)

```
Ты — senior QA engineer с опытом в логистических системах и транспортной логистике.
Твоя задача — составить структурированный чек-лист для тестирования формы.
Учитывай специфику отрасли: грузоперевозки, работа с API расчёта стоимости, UX для логистов.
Отвечай только на русском языке, используя профессиональную терминологию.
```

## Пользовательский промпт (user)

```
Мы разрабатываем MVP формы создания заявки на перевозку для сервиса «УЛ Лайт» (B2B-логистика).

Поля формы:
1. Город отправления — автокомплит / выпадающий список из базы данных.
2. Город назначения — автокомплит / выпадающий список из базы данных.
3. Вес груза — числовое поле, в килограммах, максимум 20 000 кг.
4. Объём груза — числовое поле, в кубических метрах, максимум 82 м³.
5. Дата погрузки — датапикер, нельзя выбрать прошедшую дату.
6. Кнопка «Рассчитать стоимость» — отправляет POST-запрос к API, возвращает цену в рублях или ошибку, если маршрут не найден.

Составь чек-лист для тестирования этой формы, включающий:
- Позитивные сценарии (happy path)
- Негативные сценарии (ошибки, некорректные данные)
- Граничные значения (boundary testing)
- Проверки UX и доступности
- Интеграционные проверки (API)

Для каждого пункта укажи:
- ID сценария
- Название проверки
- Приоритет (P0/P1/P2/P3)
- Конкретные шаги (что именно ввести, на что нажать)
- Ожидаемый результат

Учти, что:
- Города выбираются из реальной базы данных (не все пары городов имеют маршруты)
- Это B2B-сервис, пользователи — профессиональные логисты, важна скорость работы
- Поле «вес» должно принимать только положительные значения
- Поле «объём» не может быть отрицательным и не может быть нулевым (груз имеет ненулевой объём)
- API может быть недоступен или возвращать ошибки
- Пользователь может вводить данные быстро, с клавиатуры и мыши
```
