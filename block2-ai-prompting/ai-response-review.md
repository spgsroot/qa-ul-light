# Экспертное ревью ответа ИИ (Блок 2, ключевая часть)

## Общая оценка

**Качество ответа: 6.5/10** — ИИ сгенерировал базовый чек-лист, покрывающий очевидные сценарии, но упустил ряд критичных для логистического B2B-сервиса моментов. Чек-лист пригоден как стартовая точка, но требует существенной доработки инженером QA перед использованием.

---

## Что ИИ сделал хорошо

1. **Структурирование по категориям** — разделение на Happy Path / Negative / Boundary / UX / Integration — правильный подход
2. **Приоритизация** — P0 для блокирующих проверок, P2 для UX — корректно
3. **Учёт API-ошибок** — проверки на 404 (маршрут не найден), 500 (недоступность), 401/403 (авторизация) — хорошее покрытие
4. **Граничные значения для числовых полей** — стандартные техники (±1 от границы) применены корректно
5. **UX базового уровня** — клавиатурная навигация, индикация загрузки — то, что часто забывают

---

## Критические упущения (специфика логистики не учтена)

### 1. Нет проверок на некорректные пары «вес/объём»
**В реальной логистике** вес и объём связаны: есть понятие «плотность груза» (кг/м³). Если пользователь вводит вес = 20000 кг и объём = 0.01 м³ — это физически невозможно (плотность 2 000 000 кг/м³). Система должна выдавать предупреждение.

❌ **ИИ это полностью пропустил.** Это критичная проверка для логистического сервиса.

### 2. Нет проверок на дробные значения веса
В логистике вес часто указывается с точностью до десятых/сотых (например, 1500.5 кг). ИИ проверяет только целые граничные значения и буквы в поле, но не уточняет, какой разделитель использовать (точка или запятая), и какой шаг допустим.

### 3. Нет проверок на «пустой» автокомплит и edge-кейсы выбора города
- Что если база городов не загрузилась (ошибка API)?
- Что если ввести несуществующий город — текстовое поле позволяет свободный ввод?
- Что если у города есть несколько вариантов (Москва и Московская область)?
- Автокомплит с задержкой (debounce) — проверка времени отклика

❌ **ИИ предполагает, что автокомплит работает идеально.** В реальности это один из самых багогенных компонентов.

### 4. Нет проверок на специальные символы и инъекции
- SQL-инъекция в поле города: `Москва'; DROP TABLE cities;--`
- XSS в названии города
- Эмодзи и Unicode-символы в полях
- Очень длинные строки в полях ввода

❌ **ИИ проверяет только буквы в числовом поле, но не проверяет инъекции.**

### 5. Нет проверок на конкурентные сценарии
- Двойное нажатие кнопки «Рассчитать» (двойной клик → два запроса → двойное списание?)
- Отправка формы при медленном соединении (повторное нажатие после таймаута)
- Множественные быстрые запросы (rate limiting)

❌ **ИИ упоминает блокировку кнопки в UX-03, но не проверяет двойной клик как отдельный сценарий.**

### 6. Нет проверки кэширования расчётов
Логисты часто пересчитывают стоимость с разными параметрами. Если для пары «Москва → СПб, 500 кг, 10 м³» расчёт уже был выполнен — должен ли API кэшировать результат? Должна ли система предотвращать повторные идентичные запросы?

## Что ИИ сделал избыточно или нерелевантно

### 1. BV-09: Дата погрузки = сегодня
ИИ пишет: «Зависит от бизнес-правил, уточнить у PM». **Это правильно — не надо гадать.** Но сам кейс сформулирован как тест-кейс, хотя это вопрос к требованиям, а не тест.

👉 **Удалить из чек-листа, перенести в список вопросов к PM.**

### 2. BV-10: Дата погрузки = сегодня + 365 дней
В B2B-логистике никто не планирует перевозку за год. Горизонт планирования — от 1 до 30 дней. Проверка +365 дней — синтетический кейс, не имеющий отношения к реальности.

👉 **Заменить на проверку горизонта планирования (30-90 дней — уточнить у бизнеса).**

### 3. UX-05: Адаптивность (320px)
В B2B-сервисе для логистов мобильная версия с шириной 320px — крайне маловероятный сценарий. Логисты работают с десктопов или планшетов (min-width 768px).

👉 **Понизить приоритет до P3 или заменить на проверку планшетной версии (768-1024px).**

### 4. INT-02: Retry при ошибке сети
Retry-логика на фронтенде для POST-запроса расчёта стоимости — опасная практика без идемпотентности. Если первый запрос дошёл до сервера, но ответ потерялся, повторный запрос создаст дубликат.

👉 **Не реализовывать без идемпотентного ключа. Кейс требует уточнения архитектуры.**

## Что ИИ забыл учесть

### Специфика логистики
| # | Пропущенная проверка | Почему критично |
|---|----------------------|-----------------|
| 1 | **Плотность груза (weight/volume ratio)** | Стандартная проверка в TMS: система должна предупреждать о нереалистичных значениях |
| 2 | **Категория/класс груза** | Нет в форме MVP, но влияет на стоимость. Нужно проверить, что отсутствие поля не ломает API |
| 3 | **ADR / опасные грузы** | Для некоторых пар городов опасные грузы запрещены. Если в будущем добавят — это повлияет на логику |
| 4 | **Тип транспорта** | Вес 20000 кг на газели не повезёшь. Без выбора типа ТС невозможно валидно рассчитать стоимость |
| 5 | **Сроки доставки** | Логисту важно не только «сколько стоит», но и «когда привезут». API может возвращать срок, форма должна его показывать |

### Специфика UX для профессиональных пользователей
| # | Пропущенная проверка | Почему критично |
|---|----------------------|-----------------|
| 6 | **Горячие клавиши** | Профи используют Ctrl+Enter для отправки формы. ИИ проверил только Tab |
| 7 | **Сохранение черновика** | Логист может начать заполнять, отвлечься, вернуться. Данные не должны теряться |
| 8 | **История расчётов** | Пользователь меняет параметры и хочет видеть предыдущие результаты для сравнения |
| 9 | **Копирование параметров из предыдущей заявки** | Частый сценарий: «повторить заявку с другими датами» |

### Специфика интеграции
| # | Пропущенная проверка | Почему критично |
|---|----------------------|-----------------|
| 10 | **Формат валюты в ответе API** | Цена в рублях — но как именно? `15000`, `15000.00`, `"15 000 ₽"` — нужно проверять парсинг |
| 11 | **Обработка частичного ответа API** | Что если API вернуло цену, но без breakdown? Или только для части маршрута? |
| 12 | **Таймауты** | Нет проверки на зависший API. Пользователь не должен ждать бесконечно |

## Общие выводы

1. **ИИ хорошо справляется с типовыми паттернами** (numeric boundary, required fields, HTTP status codes), но **не понимает предметную область** (логистика, B2B).
2. **Промпт можно улучшить:** явно перечислить специфичные для логистики риски (плотность груза, типы ТС, ADR), тогда ИИ учтёт их.
3. **Без экспертной фильтрации** чек-лист ИИ создаёт ложное ощущение покрытия: 28 кейсов выглядят внушительно, но пропускают критичные отраслевые сценарии.
4. **Рекомендация:** использовать ИИ для генерации черновика, но обязательно ревьюировать senior QA с отраслевой экспертизой. Никогда не использовать сырой ответ ИИ как финальный чек-лист.
