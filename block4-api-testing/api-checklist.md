# Блок 4: Тестирование API (контроллер /api/v2/orders)

## Контекст

Контроллер `/api/v2/orders` разработан в соответствии со стандартами Google REST API Guide и Microsoft REST API Guidelines. Поддерживает стандартные CRUD-операции с логическим удалением.

## Исходная спецификация (восстановлено из OCR / Yandex Disk)

| Метод | Интерпретация (Google) | URL | Успешные статус-коды | Комментарии |
|-------|----------------------|-----|----------------------|-------------|
| `GET` | List | `/api/v2/orders` | `200 OK` | При отсутствии записей — пустой массив `[]`. Только неудалённые записи |
| `GET` | Get | `/api/v2/orders/{id:uuid}` | `200 OK`, `404 Not Found`, `410 Gone` | `410 Gone` рекомендуется при наличии `DELETE` (логическое удаление) и `PUT /.../undelete` |
| `POST` | Create | `/api/v2/orders/` или `/api/v2/orders/{id:uuid}` | `200 OK` или `201 Created`, `409 Conflict` | `201` при наличии парного `GET /{id}`. `409` для случая `POST /.../{id:uuid}` |
| `PUT` | Update | `/api/v2/orders/{id:uuid}` | `200 OK` или `204 No Content` | `200` если возвращается обновлённая read-model, иначе `204` |
| `PUT` | Create or Update (upsert) | `/api/v2/orders/{id:uuid}` | `201 Created` / `200 OK` (создание), `204 No Content` / `200 OK` (обновление) | Для синхронизации Id между системами. Предпочтительны `201`/`204` для понимания выполненного сценария |
| `DELETE` | Delete | `/api/v2/orders/{id:uuid}` | `204 No Content`, `404 Not Found`, `410 Gone` | `410` — запись была логически удалена ранее |
| `HEAD` | — | `/api/v2/orders/{id:uuid}` | `204 No Content` | Проверка существования |

## Чек-лист тестирования API

### 1. GET /api/v2/orders (List — список заказов)

| ID | Проверка | Ожидаемый статус | Тело ответа | Приоритет |
|----|----------|-----------------|-------------|-----------|
| API-01 | Получение списка при наличии заказов | `200 OK` | Массив заказов (JSON). Каждый заказ содержит все поля read-model | P0 |
| API-02 | Получение списка при отсутствии заказов | `200 OK` | `[]` (пустой массив) | P0 |
| API-03 | В ответе отсутствуют логически удалённые заказы (`isDeleted: true`) | `200 OK` | Только активные заказы | P1 |
| API-04 | Пагинация: параметры `?page=1&size=10` | `200 OK` | Массив из ≤10 элементов. Заголовки `Link` для пагинации (RFC 5988) | P1 |
| API-05 | Пагинация: `?page=0` (некорректная страница) | `400 Bad Request` | `{"error": "page must be >= 1"}` | P2 |
| API-06 | Пагинация: `?size=0` (размер страницы = 0) | `400 Bad Request` | Ошибка валидации | P2 |
| API-07 | Сортировка: `?sort=createdAt&order=desc` | `200 OK` | Заказы отсортированы по убыванию даты создания | P1 |
| API-08 | Сортировка: несуществующее поле `?sort=nonexistent` | `400 Bad Request` | `{"error": "unknown sort field: nonexistent"}` | P2 |
| API-09 | Фильтрация: query-параметры (например, `?status=delivered`) | `200 OK` | Только заказы с указанным статусом | P1 |
| API-10 | Фильтрация: несуществующее значение фильтра | `200 OK` | `[]` (пустой массив) — фильтр применён, нет совпадений | P2 |

### 2. GET /api/v2/orders/{id} (Get — один заказ)

| ID | Проверка | Ожидаемый статус | Тело ответа | Приоритет |
|----|----------|-----------------|-------------|-----------|
| API-11 | Получение существующего активного заказа | `200 OK` | Полный объект заказа (read-model) | P0 |
| API-12 | Заказ не существует (UUID никогда не создавался) | `404 Not Found` | `{"error": "Order not found", "id": "..."}` | P0 |
| API-13 | Заказ был логически удалён (`isDeleted: true`) | `410 Gone` | `{"error": "Order deleted", "id": "...", "deletedAt": "..."}` | P1 |
| API-14 | Некорректный формат UUID в пути (`/orders/abc`) | `400 Bad Request` | `{"error": "Invalid UUID format"}` | P1 |
| API-15 | ID с некорректными символами (SQL injection попытка: `/orders/1';DROP TABLE--`) | `400 Bad Request` | Ошибка валидации UUID | P1 |
| API-16 | Проверка заголовков: `Content-Type: application/json` | `200 OK` | Заголовок ответа `Content-Type: application/json` | P2 |
| API-17 | Проверка заголовков: `ETag` и `Last-Modified` (если conditional GET) | `200 OK` | Заголовки присутствуют | P2 |
| API-18 | Conditional GET: `If-None-Match` с актуальным ETag | `304 Not Modified` | Пустое тело | P2 |
| API-19 | Conditional GET: `If-None-Match` с устаревшим ETag | `200 OK` | Полный объект заказа | P2 |

### 3. POST /api/v2/orders (Create — создание заказа)

| ID | Проверка | Ожидаемый статус | Тело ответа | Приоритет |
|----|----------|-----------------|-------------|-----------|
| API-20 | Создание заказа со всеми обязательными полями (валидное тело) | `201 Created` | Созданный объект заказа. Заголовок `Location: /api/v2/orders/{newId}` | P0 |
| API-21 | Создание заказа с опциональными полями | `201 Created` | Объект заказа со всеми переданными полями | P1 |
| API-22 | Создание без обязательных полей (например, без `origin`, `destination`) | `400 Bad Request` / `422 Unprocessable Entity` | Детализация ошибки: перечень отсутствующих полей | P0 |
| API-23 | Создание с отрицательным весом (`weight: -100`) | `400 Bad Request` / `422 Unprocessable Entity` | `{"error": "weight must be positive"}` | P0 |
| API-24 | Создание с весом = 0 | `400 Bad Request` / `422 Unprocessable Entity` | Ошибка валидации (если 0 недопустим) | P1 |
| API-25 | Создание с весом > максимального | `400 Bad Request` / `422 Unprocessable Entity` | Ошибка валидации | P1 |
| API-26 | POST с указанием конкретного UUID: `/api/v2/orders/{id:uuid}` (создание по заданному ID) | `201 Created` | Заказ создан с указанным ID | P1 |
| API-27 | POST с UUID, который уже существует | `409 Conflict` | `{"error": "Order with id ... already exists"}` | P1 |
| API-28 | POST с некорректным Content-Type (например, `text/plain`) | `415 Unsupported Media Type` | Ошибка | P2 |
| API-29 | POST с некорректным JSON (синтаксическая ошибка) | `400 Bad Request` | `{"error": "Malformed JSON"}` | P1 |
| API-30 | POST с невалидным UUID в теле (ссылка на несуществующий ресурс) | `400 Bad Request` / `422 Unprocessable Entity` | Ошибка валидации ссылочной целостности | P1 |
| API-31 | POST с полем, которого нет в схеме (unknown field) | `201 Created` (игнорировать) или `400 Bad Request` | Зависит от политики API. Google: игнорировать неизвестные поля | P2 |

### 4. PUT /api/v2/orders/{id} (Update — обновление)

| ID | Проверка | Ожидаемый статус | Тело ответа | Приоритет |
|----|----------|-----------------|-------------|-----------|
| API-32 | Полное обновление всех полей заказа (Full Update) | `200 OK` (с read-model) или `204 No Content` | В зависимости от реализации | P0 |
| API-33 | Частичное обновление (Partial Update) — только изменившиеся поля | `200 OK` | Обновлённый объект | P1 |
| API-34 | Обновление несуществующего заказа | `404 Not Found` | Ошибка | P0 |
| API-35 | Обновление логически удалённого заказа | `410 Gone` | `{"error": "Order deleted"}` | P1 |
| API-36 | Обновление с некорректными данными (вес = -100) | `400 Bad Request` / `422 Unprocessable Entity` | Ошибка валидации | P0 |
| API-37 | PUT как Create-or-Update (upsert): создание нового заказа с заданным UUID | `201 Created` | Созданный объект. Заголовок `Location` | P1 |
| API-38 | PUT как Create-or-Update: обновление существующего | `200 OK` или `204 No Content` | Обновлённый объект или пустое тело | P1 |
| API-39 | Обновление с ID в теле, не совпадающим с ID в URL (если поле id присутствует в теле) | `400 Bad Request` | `{"error": "ID in body does not match URL"}` | P1 |
| API-40 | PUT с конфликтующими данными (нарушение уникальности) | `409 Conflict` | Ошибка конфликта | P1 |

### 5. DELETE /api/v2/orders/{id} (Delete — логическое удаление)

| ID | Проверка | Ожидаемый статус | Тело ответа | Приоритет |
|----|----------|-----------------|-------------|-----------|
| API-41 | Удаление существующего активного заказа | `204 No Content` | Пустое тело (или `200 OK` с подтверждением) | P0 |
| API-42 | После удаления — заказ не возвращается в GET / (List) | `200 OK` в списке | Удалённый заказ отсутствует в выдаче | P0 |
| API-43 | После удаления — GET /{id} возвращает `410 Gone` | `410 Gone` | Информация о дате удаления | P0 |
| API-44 | Повторное удаление того же заказа | `410 Gone` | `{"error": "Order already deleted"}` | P1 |
| API-45 | Удаление несуществующего заказа | `404 Not Found` | Ошибка | P1 |
| API-46 | Мягкое удаление (soft delete) — проверка поля `deletedAt` / `isDeleted` в БД | Запись не удалена физически | `isDeleted = true`, `deletedAt = NOW()` | P1 |

### 7. Обработка ошибок (общие сценарии)

### 8. Идемпотентность и конкурентность

| ID | Проверка | Ожидаемый статус | Приоритет |
|----|----------|-----------------|-----------|
| API-58 | Повторный POST с одним `Idempotency-Key` | Тот же результат, что при первом запросе | P1 |
| API-59 | Параллельные PUT с разными данными (optimistic locking через `If-Match` / ETag) | `412 Precondition Failed` для второго запроса (конфликт версий) | P1 |
| API-60 | PUT без `If-Match` (если требуется) | `428 Precondition Required` | P2 |

## Сводка ожидаемых статус-кодов

| Сценарий | Ожидаемые коды |
|----------|---------------|
| Успешное получение / создание / обновление | `200`, `201`, `204` |
| Ошибки валидации (некорректные данные) | `400`, `422` |
| Аутентификация (нет/истёк токен) | `401` |
| Авторизация (нет прав) | `403` |
| Несуществующий ID | `404` |
| Логически удалённая запись | `410` |
| Конфликт (дубликат, состояние) | `409` |
| Precondition (версия) | `412`, `428` |
| Неподдерживаемый метод | `405` |
| Неподдерживаемый Content-Type | `415` |
| Rate limiting | `429` |
| Падение БД / зависимостей | `503` |
| Таймаут | `504` |
